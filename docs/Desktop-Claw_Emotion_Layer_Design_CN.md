# Desktop-Claw 情绪层设计稿

## 1. 文档目标

本文档用于定义 Desktop-Claw 第一阶段“情绪层”的产品设计与技术实现方向。

这不是一个纯视觉换皮方案，也不是一个完整桌宠动画系统。
它的目标是：

- 让 Claw 在不聊天时也有“活着”的感觉
- 让悬浮球从功能入口升级为有状态的桌面存在
- 为后续记忆层、主动性、换皮系统预留清晰接口

本文档重点覆盖：

- 状态机设计
- 数据模型
- 后端接口
- 前端表达规则
- 交互规则
- MVP 边界
- 分阶段实现路线

---

## 2. 设计目标

### 产品目标

用户在不开面板、不输入文字的情况下，也能感知到：

- Claw 有状态
- Claw 会变化
- Claw 会对互动产生不同反应
- Claw 会随着时间和相处节奏表现出不同情绪

### 体验目标

第一版情绪层上线后，用户应该能明显感受到：

- 白天和深夜的 Claw 不一样
- 刚聊完和很久没聊的 Claw 不一样
- 今天聊得很多、现在安静下来后的 Claw 不一样
- 点一下它时，不同状态下的反馈不同

### 工程目标

第一版不要依赖复杂动画框架，也不要依赖 LLM 的随机情绪判断。

第一版必须具备：

- 一个稳定、可解释的状态机
- 一个统一的情绪数据源
- 一个前后端共享的状态协议
- 一个可扩展的表达映射层

---

## 3. 非目标

以下内容明确不属于第一版情绪层范围：

- 动漫形象、Live2D、骨骼动画
- 大量服装、家具、物品系统
- 饥饿值、清洁值、惩罚式养成
- LLM 驱动的自由情绪生成
- 高频主动打扰
- 游戏化任务系统

这些能力未来可以做，但不应该阻塞第一版情绪层。

---

## 4. 设计原则

### 原则 1：情绪首先是状态，不是装饰

情绪层不是给悬浮球加一点动效，而是为 Claw 提供一套统一状态模型。

### 原则 2：克制优先

Claw 要有生命感，但不能吵、不能跳、不能乱闪、不能影响工作注意力。

### 原则 3：可解释优先

每一个状态变化都应该有明确来源，便于调试、埋点和后续优化。

### 原则 4：先做确定性，再做复杂性

第一版只做规则驱动状态机，不做开放式情绪推理。

### 原则 5：为未来留接口

后续记忆层、主动性、换皮系统都要能接入情绪层，而不是推翻重来。

---

## 5. 系统定位

情绪层在整体架构中的作用是：

- 输入：时间、互动频率、任务状态、会话节奏、后续可扩展的记忆信号
- 处理：根据状态机规则推导当前情绪状态
- 输出：提供给前端做视觉表达、气泡文案、交互反馈、主动行为节奏控制

可以把它理解为：

**Claw 的“当前状态引擎”**

---

## 6. 第一版状态模型

### 6.1 状态定义

第一版建议保留 4 个主状态：

#### 1. idle

含义：
空闲、在场、轻陪伴。

用户感知：
Claw 在，但不忙、不困、不收尾。

#### 2. busy

含义：
近期刚与用户频繁互动，处于活跃陪伴状态。

用户感知：
Claw 正在“投入”和你相处，有精神、有回应感。

#### 3. done

含义：
今天已经有较多互动，现在进入一个轻松、收工、满意的安静状态。

用户感知：
像“今天陪你待了挺久，现在安静地歇着”。

#### 4. night

含义：
深夜模式，安静、低亮、困倦、收敛。

用户感知：
它还在，但明显不想太活跃，会更克制、更柔和。

## 7. 第一版状态机规则

### 7.1 输入信号

第一版建议只使用以下输入：

- 当前时间
- 今日消息记录
- 最近一条消息时间
- 当前是否在流式回复或刚完成任务

这些信息现有项目基本已经具备，尤其是：
[emotion.ts](/Users/djtang/Code/workshop/Desktop-Claw/packages/backend/src/memory/emotion.ts)
[memory-service.ts](/Users/djtang/Code/workshop/Desktop-Claw/packages/backend/src/memory/memory-service.ts)

### 7.2 基础规则

建议状态优先级：

`night > busy > done > idle`

也就是说，先判断是否进入夜间模式，再判断是否活跃，再判断是否是完成后的收尾状态，最后默认回到 idle。

### 7.3 具体规则定义

#### 规则 A：night

触发条件：

- 本地时间在 22:00 到次日 06:00 之间

进入后行为：

- 视觉更柔和、更低亮
- 冒泡频率下降
- 点击反馈更轻、更短
- 不主动出现活跃状态表现

#### 规则 B：busy

触发条件：

- 最近 5 分钟内有对话
- 或当前正在流式回复
- 或最近 2 分钟内刚完成一次任务

进入后行为：

- 视觉更有精神
- 对点击和 hover 响应更快
- 问候语更像“在呢”“继续说”

#### 规则 C：done

触发条件：

- 今日累计用户消息数 >= 5
- 最近 30 分钟内没有新对话
- 且不满足 busy

进入后行为：

- 视觉更松弛、更稳定
- 气泡更像收工、陪伴、休息
- 减少过强反馈

#### 规则 D：idle

触发条件：

- 不满足 night、busy、done

进入后行为：

- 平稳待机
- 有轻微存在感
- 可在适度随机区间内生成小气泡

## 8. 状态切换策略

### 8.1 切换原则

状态切换不能太频繁，否则会显得混乱和假。

建议采用：

- 基于规则的即时判断
- 叠加最小保持时长

### 8.2 最小保持时长

建议值：

- busy：最少保持 2 分钟
- done：最少保持 10 分钟
- idle：无强保持约束
- night：由时间段直接决定

### 8.3 切换去抖

建议增加去抖机制：

- 同一状态短时间内重复计算，不重复广播
- 切换频率不超过每 30 秒一次

## 9. 数据模型设计

### 9.1 EmotionState

```ts
type EmotionState = 'idle' | 'busy' | 'done' | 'night'
```

### 9.2 EmotionSnapshot

建议后端统一输出结构：

```ts
interface EmotionSnapshot {
  state: EmotionState
  reason: string
  updatedAt: string
  phrases: string[]
  intensity: number
  ambientLevel: 'low' | 'medium' | 'high'
}
```

### 9.3 字段说明

- `state`
  当前主状态

- `reason`
  状态来源，供调试和日志使用
  例如：
  `night_time`
  `recent_conversation`
  `quiet_after_active_day`
  `default_idle`

- `updatedAt`
  最近一次状态变更时间

- `phrases`
  当前状态下适合单击、冒泡、轻反馈的话术池

- `intensity`
  情绪表现强度，范围建议 0 到 1
  第一版可以固定几个值：
  `night=0.2`
  `idle=0.4`
  `done=0.5`
  `busy=0.8`

- `ambientLevel`
  用于控制冒泡频率和视觉活动度

## 10. 后端实现方案

### 10.1 结构建议

建议将当前的 [emotion.ts](/Users/djtang/Code/workshop/Desktop-Claw/packages/backend/src/memory/emotion.ts) 从纯函数升级为两层：

#### 层 1：纯规则推导

保留纯函数能力：

```ts
deriveEmotionState(...)
```

职责：

- 输入基础信号
- 输出推导结果

#### 层 2：EmotionService

新增服务对象：

```ts
class EmotionService
```

职责：

- 维护当前 emotion snapshot
- 处理状态刷新
- 提供接口读取
- 控制状态广播
- 管理 phrases、强度、活动度

### 10.2 输入来源

EmotionService 第一版可从以下地方拿数据：

- `memoryService.getTodayMessages()`
- 当前时间
- task coordinator / ws 层的活跃事件

建议额外维护两类运行态信号：

- `lastInteractionAt`
- `lastTaskCompletedAt`

### 10.3 刷新时机

建议在以下时机刷新情绪状态：

- 应用启动后初始化一次
- 每 60 秒定时刷新一次
- 用户发送消息时刷新
- 流式回复开始时刷新
- 任务完成时刷新
- 跨越时间段边界时刷新

### 10.4 对外接口

第一版建议提供：

#### HTTP

```http
GET /emotion
```

返回：

```json
{
  "state": "idle",
  "reason": "default_idle",
  "updatedAt": "2026-04-02T12:00:00.000Z",
  "phrases": ["嗨～有什么想聊的吗？"],
  "intensity": 0.4,
  "ambientLevel": "medium"
}
```

#### 可选 WS 事件

如果要做得更顺，第二步可加：

```ts
type: 'companion.emotion_changed'
```

第一版若不想改 WS 协议，也可以先轮询 HTTP。

## 11. 前端表现设计

### 11.1 总体原则

前端不要直接计算状态，只消费后端给出的 emotion snapshot。

前端职责是：

- 把状态映射为视觉表现
- 把状态映射为交互反馈
- 把状态映射为冒泡节奏

### 11.2 视觉表达维度

建议悬浮球的视觉表达拆成五个维度：

#### 1. 发光强度

- night：低
- idle：中低
- done：中
- busy：中高

#### 2. 呼吸节奏

- night：慢、轻
- idle：平稳
- done：更松弛
- busy：更明显、更有活力

#### 3. 阴影和边缘扩散

- busy：更明显
- night：更弱

#### 4. 微动作幅度

- night：几乎无
- idle：轻微
- done：轻微偏松
- busy：略明显

#### 5. 色温或亮度

即使仍然是悬浮球，也可以通过亮度和色温做状态感。
不建议第一版做大色彩跳变，优先做细节差异。

### 11.3 自动冒泡

第一版可以做有限自动冒泡，但必须克制。

建议规则：

- 只在 `idle` 和 `done` 下允许少量环境式冒泡
- `busy` 下主要依赖点击反馈和任务状态
- `night` 下大幅减少自动冒泡

建议节奏：

- idle：6 到 15 分钟随机一次
- done：10 到 20 分钟随机一次
- busy：默认不自动冒泡
- night：默认不自动冒泡，除非用户主动点击

自动冒泡必须满足：

- 当前没有流式状态
- 当前没有打开 QuickInput
- 用户最近没有显式关闭气泡

## 12. 交互规则设计

### 12.1 点击反馈

点击悬浮球时，反馈应随状态变化。

#### idle

感觉：
轻松、愿意搭话

反馈风格：

- “在呢～”
- “今天怎么样？”
- “想聊聊吗？”

#### busy

感觉：
投入、在线、精神

反馈风格：

- “我在听～”
- “还有什么要继续吗？”
- “嗯？我在呢”

#### done

感觉：
松弛、陪伴、收工感

反馈风格：

- “今天也待了挺久呢”
- “辛苦啦，我还在”
- “休息一下也好”

#### night

感觉：
困倦、安静、低打扰

反馈风格：

- “夜深了，早点休息哦”
- “我还在，不过今天别太晚啦”
- “困困的陪着你”

### 12.2 Hover 反馈

建议第一版不要做过度 hover 动作，只做很轻的响应：

- 停止穿透后稍微提亮
- 呼吸幅度略增强
- 可选非常轻的阴影变化

### 12.3 拖拽反馈

拖拽时不必做复杂表现，但拖拽结束可以有一个轻量反馈：

- busy：更快恢复活跃状态
- idle：平稳恢复
- night：较弱恢复

### 12.4 QuickInput 反馈

打开 QuickInput 时：

- 视为一次显式主动互动
- 可临时将状态拉向 busy
- 关闭后不立刻退出 busy，而是按规则自然衰减

## 14. 与记忆层的接口预留

虽然第一版情绪层不深度依赖记忆，但要留接口给未来接入。

后续可引入的记忆信号包括：

- 最近几天的互动密度
- 是否有共同纪念日
- 是否有进行中的重要主题
- 某些用户偏好的互动方式
- 某些需要特殊对待的日子

建议后端 EmotionService 未来保留扩展输入：

```ts
interface EmotionSignals {
  now: Date
  todayMessages: PersistedMessage[]
  lastInteractionAt?: string
  lastTaskCompletedAt?: string
  memoryHints?: string[]
  specialDayTags?: string[]
}
```

## 15. 埋点与验证

第一版建议至少记录以下指标：

- 每日点击悬浮球次数
- QuickInput 打开次数
- 面板打开次数
- 自动冒泡被关闭的频率
- 不同情绪状态停留时长
- 用户活跃日里各状态分布

体验验证建议：

- 用户是否能主动说出状态差异
- 用户是否觉得打扰感增强
- 用户是否更愿意让 Claw 常驻

## 16. MVP 边界

第一版必须做：

- 四状态状态机
- `/emotion` 接口
- 悬浮球状态化视觉差异
- 点击反馈按状态变化
- 克制的自动冒泡策略

第一版可以不做：

- WS 广播
- 大量互动手势
- ChatPanel 状态展示
- Profile 中的情绪展示
- 复杂动画资产
- 个性化外观皮肤

## 18. 关键风险

### 风险 1：表现太弱，用户感知不到

解决方式：

- 保证四种状态至少有明显但克制的差异
- 不要只改一个 opacity

### 风险 2：表现太强，影响工作

解决方式：

- 禁止高频闪烁
- 限制冒泡频率
- 降低 hover 和 idle 动作幅度

### 风险 3：状态切换太频繁，显得假

解决方式：

- 增加最小保持时长
- 增加状态切换去抖

### 风险 4：后端状态和前端表现不一致

解决方式：

- 前端只消费 snapshot，不重复推导
- 所有状态逻辑都收口到后端

---

## 19. 最终判断

情绪层不是“可爱化工程”，而是 Desktop-Claw 作为 Companion-first 产品的第一层产品骨架。

它解决的是：

- Claw 看起来是不是活的
- Claw 在前台是不是有存在感
- Claw 的行为是不是开始具备连续性

这一步一旦成立，后续记忆层前台化就会更有意义，因为用户会开始自然地相信：

**这个东西不是按钮，而是一个持续存在的桌面伙伴。**
