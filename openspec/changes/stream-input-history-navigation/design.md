## Context

Claude Code 的 Stream/Stdio 协议（NDJSON）完整消息清单中不含任何 `history` / `input_history` / `prompt_history` 类型。因此，通过 Stream 接入的外部客户端需要**自行维护历史输入**，实现 ↑/↓ 键导航功能。

当前状态：客户端无历史输入管理能力，每次会话独立。

## Goals / Non-Goals

**Goals:**
- 实现本地历史输入存储和导航功能
- 支持 ↑/↓ 键翻阅历史记录
- 草稿保存和恢复机制
- 历史条目上限保护（100 条）

**Non-Goals:**
- 修改 Claude Code Stream 协议
- 跨设备历史同步
- 历史搜索/过滤功能
- 与 Claude Code 内置历史集成

## Decisions

### 决策 1：客户端本地维护历史

**选择**：由客户端自行实现 `InputHistory` 类，不依赖服务端。

**理由**：
- Stream 协议未暴露历史相关消息类型
- 本地实现响应更快，无需网络往返
- 与实现语言无关，任何客户端语言均可实现

**替代方案**：扩展 Stream 协议添加历史消息类型 — 拒绝，需修改 Claude Code 源码，维护成本高。

### 决策 2：LIFO 倒序存储

**选择**：新条目 `unshift` 到数组头部，最新记录在索引 0。

**理由**：
- 翻阅历史时 ↑ 键对应索引增加（向前翻阅更早记录）
- 与终端 Shell 历史行为一致

### 决策 3：草稿保存机制

**选择**：首次按 ↑ 时保存当前未提交输入，按 ↓ 回到底部时恢复。

**理由**：
- 参考 Claude Code 内部实现 `useArrowKeyHistory.tsx`
- 用户常见的误操作恢复需求

### 决策 4：持久化作为可选扩展

**选择**：基础 `InputHistory` 类不依赖文件系统，额外提供 `PersistentInputHistory` 子类。

**理由**：
- 部分客户端（如 Web 版）可能无文件系统访问权限
- 保持核心逻辑简洁，持久化按需引入

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 历史记录丢失（重启后） | 可选持久化存储，使用 JSONL 格式追加写入 |
| 内存无限增长 | 设置 maxItems = 100，超出时裁剪 |
| 多客户端历史不同步 | 文档说明此为客户端本地功能，非同步功能 |
| 并发写入冲突 | 持久化使用追加模式，避免覆盖 |
