## Context

悬浮球是 Coding-bubble 在用户桌面上常驻的唯一可见元素。主窗口关闭时，用户需要通过悬浮球了解 Claude Code 会话状态。当前悬浮球仅在干预（权限请求/等待输入）时显示气泡通知，无法持续反映会话运行状态。

`state-machine-redesign` 变更已在 SessionStore 中实现了 `resolveDisplayState()` 方法，可返回所有 session 的最高优先级状态。本变更将其连接到悬浮球的视觉指示器。

## Goals / Non-Goals

**Goals:**
- 在悬浮球右下角添加状态圆点，显示所有 session 的最高优先级状态
- 仅在主面板关闭且存在非 idle/ended 活跃会话时显示
- 各状态有对应的颜色和动画效果

**Non-Goals:**
- 悬浮球情绪层（data-emotion）激活 — 不在本次范围
- 改变现有 badge 逻辑（右上角红色圆点）— 保持不变
- 在主面板打开时也显示状态圆点 — 不需要

## Decisions

### D1: 圆点位置 — 右下角

**选择：** 球体右下角，与右上角的 badge 错开

**替代方案：** 球体底部中央

**理由：** 右上角已被 badge 占用。右下角视觉上自然，且不遮挡球体中心图标。

### D2: 数据传输 — 复用现有 IPC 事件

**选择：** 在 `bubbleControllerSync` 中通过新的 `bubble:status` 事件发送 displayState 字符串

**替代方案：** 在 `bubble:show`/`bubble:hide` 事件中附加状态信息

**理由：** 独立事件更清晰，状态变化和通知显示是两个独立的关注点。`bubble:status` 可以独立触发（即使没有通知需要显示）。

### D3: 圆点大小 — 8px

**选择：** 8px 直径，与 badge (10px) 略小，表示次要信息

**理由：** badge 表示紧急干预（需要用户操作），状态圆点表示信息性状态。大小差异帮助用户区分优先级。

## Risks / Trade-offs

**[圆点与 badge 视觉冲突]** → 两个圆点在不同位置（右上/右下），颜色也不同（badge 红色/状态圆点多色）。**缓解：** 间距足够，不会重叠。

**[频繁状态切换导致圆点闪烁]** → 状态变化太快时圆点可能频繁变色。**缓解：** 可在 CSS 层加 `transition` 平滑切换颜色（0.3s），不需要额外的防闪切逻辑。
